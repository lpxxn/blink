# Blink 项目总览

## 项目定位

Blink 是一个轻量、高性能的微博客平台，目标是支持用户发布短内容、日常动态和视觉内容。后端已按 DDD 分层落地多条用例（帖子、评论、管理、站内通知等），并在领域层约定了异步通知相关事件名与载荷；分层与数据流图示见 [`docs/architecture.md`](../../docs/architecture.md)，规则见 `01-architecture.md`，通知管线见 `docs/watermill-notifications.md`。

## 产品范围

本项目目标接近 Twitter / X 一类内容平台，优先支持以下能力：

- 用户注册、登录、身份认证与权限控制
- 发布短文、想法、图片、视频等内容
- 内容查询、时间线、详情页、用户主页等读取场景
- 内容审核与敏感内容处理
- 面向不同客户端提供 REST 与 GraphQL 两套接口

## 当前阶段的技术方向

- 后端使用 Go 开发。
- 架构采用 DDD 分层设计。
- 数据库策略采用“多数据库兼容、SQLite 先实现”。
- 可观测性优先，日志需要详细且便于排障。
- 设计上优先可扩展、可测试、可维护，而不是一次性堆实现。

## 当前代码现状

- 语言与模块：Go 1.26.1，模块名为 `github.com/lpxxn/blink`。
- 启动入口：`cmd/main.go`。
- 架构分层：
  - `application/`：用例与端口（含 `eventing.NotificationPublisher` 等）。
  - `domain/`：领域模型、仓储接口与领域事件约定（`domain/event/`）。
  - `infrastructure/`：HTTP API、持久化、Watermill + Redis Stream 通知管线等。
  - `web/`、`front/`：Web 与前端相关资源（以仓库实际目录为准）。
- 领域事件：通知链路的 JSON `type` 与发布/消费实现已对齐；`domain/event/events.go` 中的点分名与 Payload 可作为与其它消费者统一的语义基准，演进时需与 Stream 协议同步。

## 规则优先级

在为本仓库编写代码、文档、脚本或自动化变更时，优先遵循以下顺序：

1. `00-overview.md`：先理解项目目标与边界。
2. `01-architecture.md`：再判断代码应该落在哪一层。
3. `02-tech-stack.md` ~ `10-devops.md`：根据主题约束实施。

## 基本原则

- 以 DDD 分层为中心组织代码，避免跨层耦合。
- 先定义清晰的领域概念，再补充接口、存储和交付层。
- 所有新增实现都应与现有骨架保持一致，不要绕过目录边界。
- 文档与规则应区分「当前已实现行为」与「约定/改进项」（例如通知发布的错误处理、消费幂等等）。

## 推荐的功能落位

- 业务用例编排：`application/`
- 核心领域模型与领域服务：`domain/`
- 持久化、外部服务适配、传输协议实现：`infrastructure/`
- Web / App / 管理台等前端实现：`front/`
- 可执行程序入口：`cmd/`

## 贡献要求

- 不要因为当前代码少，就把所有逻辑直接写进 `cmd/main.go`。
- 不要在 `infrastructure/` 中定义业务规则。
- 不要让 DTO、数据库模型、HTTP 请求对象直接污染领域对象。
- 文档和代码都应偏向“可演进结构”，而不是一次性脚手架式堆叠。
- 规则文件应尽量写成可执行约束，例如指定库、目录落位、错误结构、测试方式，而不是只写抽象口号。
