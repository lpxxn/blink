# 代码规范

## 通用原则

- 代码应追求清晰、可维护、可测试，而不是炫技。
- 先写简单、明确的实现，再考虑抽象。
- 抽象必须服务于当前真实需求，不做空洞封装。

## Go 代码风格

- 遵循 Go 社区惯例与 `gofmt` 输出。
- 使用短小包名，避免含糊或缩写过度。
- 导出符号仅在跨包确有需要时提供。
- 错误处理必须显式，不忽略返回值。
- 尽量返回带上下文的信息，但不要吞掉原始错误。

## 包与文件组织

- 一个文件只承载一组强相关职责。
- 包名应体现业务语义或技术职责。
- 避免创建 `util`, `common`, `misc` 这类无边界包。
- 公共逻辑优先放在有明确语义的位置，而不是“工具箱”。

## 结构体与接口

- 接口应靠近使用方定义，除非它就是领域契约。
- 结构体字段命名应准确表达业务含义。
- 避免为了 mock 而提前抽象一切。
- 小接口优于大而全接口。

## 函数设计

- 单个函数只做一件事。
- 参数过多时优先重构数据结构，而不是继续堆参数。
- 尽量避免隐式副作用。
- 复杂流程可拆成私有辅助函数，但不要为拆分而拆分。

## 注释要求

- 只有在代码无法自解释时才添加注释。
- 注释应解释“为什么”，而不是机械复述“做了什么”。
- 对外导出的复杂类型和核心行为建议补充文档注释。

## 错误处理

- 不允许静默失败。
- 不允许无上下文地返回裸错误字符串。
- 不要使用宽泛兜底隐藏问题。
- 需要用户可见时，应把错误转换为明确的接口层响应。

## 文档与命名语言

- 代码标识符使用英文。
- 文档可使用中文，但术语需统一。
- 业务名词应在整个项目中保持一致，例如 `Blink`, `Post`, `Timeline`, `User`。
