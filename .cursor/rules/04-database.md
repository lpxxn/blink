# 数据库规范

## 当前状态

仓库中尚未看到具体数据库实现，但 `infrastructure/persistence/` 与 `infrastructure/repository/` 已为持久化预留目录。数据库相关实现应从一开始就与领域模型保持解耦。

## 数据库策略

- 默认先实现 SQLite，便于本地开发、测试和快速落地。
- 架构上必须保留切换到 MySQL 或 PostgreSQL 的能力。
- 不允许把仅某个数据库独有的特性写死在应用层或领域层。
- 如必须使用方言特性，应将差异封装在基础设施层，并标注适用数据库。

## 分层规则

- 领域层不感知表结构、ORM 标签、SQL 方言。
- 应用层依赖仓储接口，不依赖数据库客户端。
- 基础设施层负责：
  - 连接初始化
  - 事务管理细节
  - SQL / ORM 映射
  - 仓储实现

## 建模原则

- 先基于领域概念建模，再映射到数据库。
- 聚合边界优先于表结构便利性。
- 持久化模型可以为查询优化调整，但不能反向污染领域模型。
- 主键、唯一约束、索引应服务于业务语义与查询路径。

## 仓储规则

- 在核心层定义仓储接口，在基础设施层实现。
- 仓储返回领域对象或应用层所需结果，不返回数据库框架对象。
- 查询职责与写入职责可以分离，尤其是读写模型差异明显时。
- SQLite、MySQL、PostgreSQL 的差异应通过 repository / persistence 实现层隔离。
- 事务能力、分页策略、模糊查询、时间字段处理要优先考虑跨数据库兼容性。

## 迁移与变更

- 所有 schema 变更都应可追踪、可回滚、可审查。
- 禁止手工修改生产结构而不留下迁移记录。
- 重大数据变更应附带数据修复或回填策略。

## SQL 与性能

- 明确索引策略，不依赖“以后再优化”。
- 避免 N+1 查询。
- 查询字段按需选择，不默认 `SELECT *`。
- 分页接口必须使用稳定排序。

## 数据一致性

- 明确事务边界，优先在应用层用例中定义一致性需求。
- 对幂等写入、重复提交、并发更新提前设计约束。
- 删除策略需明确：物理删除还是软删除，不可混用无说明。

## 内容审核与删除建议

对于用户发布内容，默认支持审核与可恢复治理：

- 建议保留内容状态字段，例如：`normal`, `hidden`, `sensitive`, `deleted`。
- 对“删除”操作要区分：
  - 用户前台不可见的隐藏
  - 后台审核下架
  - 物理删除
- 审核结果、操作人、操作时间应具备可追踪性。

## 审计建议

如果后续实现用户内容、关系、权限等核心数据，建议预留：

- 创建时间
- 更新时间
- 操作者信息（如适用）
- 可审计的重要状态变更记录
