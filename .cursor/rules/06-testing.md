# 测试策略

## 当前状态

仓库中暂未看到测试文件，这意味着后续新增业务代码时必须同步补足测试，而不是等功能堆积后再补。

## 测试分层

优先建立以下层次：

- 领域层单元测试：验证核心业务规则。
- 应用层测试：验证用例编排、事务、权限、仓储交互。
- 基础设施集成测试：验证数据库、HTTP、gRPC、外部服务适配。
- 端到端测试：覆盖关键用户路径。

## 测试优先级

最优先测试：

1. 领域规则
2. 易出错的状态变更
3. 鉴权和权限判断
4. 持久化读写一致性
5. API 合约

## 编写原则

- 每个测试应只验证一个明确行为。
- 测试名应体现业务语义。
- 避免只测“调用成功”，要验证输出、状态和副作用。
- 不依赖不稳定时间、随机数、网络或共享环境。

## 推荐测试策略

- 单元测试应尽量不依赖 Docker，优先使用内存替身、fake repository、test helper 保持速度。
- 数据库、缓存、消息队列等真实依赖的验证应归入集成测试。
- 集成测试可使用 Docker / Testcontainers / docker compose 等方式提供隔离环境。
- 不要把依赖 Docker 的测试也叫“单元测试”。

## Mock 与替身

- 优先测试真实领域逻辑。
- 仅在隔离外部依赖时使用 mock / fake / stub。
- 不要因为过度 mock 而把测试写成实现细节快照。

## 集成测试建议

- 数据库相关逻辑至少应有集成测试。
- API 层应验证请求绑定、状态码、错误结构和关键字段。
- 如后续引入 GraphQL，应验证 schema 行为与 resolver 权限。
- SQLite 实现需要真实验证迁移、事务、索引和查询行为。
- 如果后续支持 MySQL / PostgreSQL，至少要为公共 repository 契约准备兼容性测试。

## 回归要求

- 修复 bug 时，优先补一个能稳定复现问题的测试。
- 新增功能时，最少补齐对应层级的核心测试。
- 没有测试覆盖的复杂逻辑，不应长期停留在主分支。
