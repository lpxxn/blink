# 部署与运维规则

## 当前状态

仓库中暂未看到明确的 CI/CD、容器编排或部署脚本配置，因此本规则定义的是后续工程化建设应遵循的方向。

## 构建与发布

- 构建应可重复、可追踪、可审计。
- 发布产物应与源码版本建立清晰映射。
- 不要依赖手工步骤完成核心发布流程。

## 配置管理

- 配置与代码分离。
- 不同环境使用不同配置源，不靠手动改代码切换。
- 所有敏感配置通过安全注入方式提供。

## 环境约束

- 本地、测试、预发、生产环境的差异要尽量显式化。
- 不允许“只在某台机器能跑”的隐式依赖。
- 服务启动必须包含健康检查、超时、退出处理等基础能力。

## CI/CD 建议

后续引入流水线时，至少包含：

- 代码格式化/静态检查
- 单元测试
- 构建验证
- 制品归档
- 部署审批或自动发布策略
- 集成测试或关键依赖验证
- 数据库迁移检查

## 运维要求

- 服务必须有基础监控和告警。
- 关键发布支持快速回滚。
- 线上问题应能通过日志、指标、追踪快速定位。
- 数据库变更应纳入发布流程，而不是临时手工执行。
- 建议逐步补充自动化部署脚本，而不是长期依赖手工发布。
- 建议为服务提供健康检查、就绪检查和基础监控面板。

## 运行可靠性

- 启动失败必须快速暴露，不允许静默降级为“看似成功”。
- 外部依赖不可用时要有明确失败策略。
- 资源关闭、信号处理、优雅退出应作为服务默认能力。

## 文档要求

- 新增部署方式时，同步补充运行说明。
- 新增运维依赖时，写明职责、入口、配置项和故障排查方法。
- `docs/` 目录应用于沉淀开发文档、设计说明、领域建模和部署文档。
- 面向用户或前端的 API 文档应保持与实现同步更新。
